1. **方案核对**：阅读 `github-native-agent-orchestrator-自动处理-issue-方案.md` 全文，对比代码中实现的模块边界、角色设置及工作流转。
2. **合同与 Schema 核对**：比对 `docs/contracts/` 下的 README、data-contracts、task-state-contracts、security-and-audit-contracts、github-artifact-templates 以及 13 个 JSON Schema 与 `src/contracts/`、`src/state/`、`src/security/`、`src/policy/` 等实际代码。
3. **API设计核对**：比对 `docs/api-design/` 01-08 章节描述的协议、CLI、本地管理/查询接口与 `src/cli.ts`、`src/ui/` 等实现。
4. **审计定位**：针对所有状态、事件、错误码、方法参数进行精确的静态交叉检查。

## 1.3. 高度一致的项（实现与方案/合同一致）

### 1.3.1. 状态机 (StateMachine)
* **状态定义**：`src/state/state-machine.ts` 中定义的 15 个 `WorkflowState`（`new`, `planning`, `plan_reviewing`, `implementing`, `pr_opened`, `pr_reviewing`, `ci_waiting`, `fixing`, `merge_ready`, `merged`, `issue_closed`, `paused`, `blocked`, `failed`）与合同 `task-state-contracts.md` 完全一致。
* **状态转换**：`stateTransitions` 转换表中的 19 条规则完全匹配合同规定的Allowed Transitions转换矩阵与守卫条件。

### 1.3.2. 模块接口与方法 (Module Interfaces)
* **GitHub API Adapter**：`src/github/api.ts` 中的 `GitHubApiAdapter` 接口实现的 10 个核心方法（`createOrUpdateIssueComment`, `setIssueLabels`, `createBranch`, `commitChanges`, `createOrUpdatePullRequest`, `submitPullRequestReview`, `readCheckSummary`, `mergePullRequest`, `deleteBranch`, `closeIssue`）与 API 设计文档中描述的方法签名、幂等键以及 Request Hash 传递机制 100% 对齐。
* **Workspace Manager**：`src/workspace/manager.ts` 的方法（`createWorkspacePlan`, `validateControlledWorkspace`, `prepareImplementerWorkspace`）与设计文档的定义一致。

### 1.3.3. 错误码与错误分类 (Error Codes)
* `src/errors.ts` 中注册的 37 个错误码（如 `WEBHOOK_SIGNATURE_INVALID`, `STALE_HEAD_SHA`, `POLICY_DENIED_PATH`, `LEASE_CONFLICT` 等）与 `06-error-codes-and-permission-actions.md` 的错误列表完全匹配，且 retry/no-retry 分类行为一致。

### 1.3.4. 数据库 DDL (Database DDL)
* `src/state/sqlite-store.ts` 中创建的四张表 `deliveries`、`workflow_runs`、`state_transitions`、`idempotent_actions` 其字段名称、主外键关系、唯一性约束（特别是 `(repo_owner, repo_name, issue_number)`）与 `data-contracts.md` 高度对齐。

### 1.3.5. 运行安全约束 (Security Controls)
* **密钥脱敏**：`src/security/redaction.ts` 和 `src/contracts/validation.ts` 中实现的 `isSecretEnvKeyName`、`redactSecretLikeValues` 逻辑符合安全合同中对敏感环境变量过滤与令牌抹除的要求。
* **路径拦截**：`src/policy/path-policy.ts` 完美实现了 `allow`、`deny`、`high_risk` 三级路径评估机制。

## 1.4. 中度一致的项（结构基本正确，但存在微小差异）

### 1.4.1. 状态 Label 的简化与映射
* **方案与合同要求**：定义了包括 `agent:planning`, `agent:plan-review`, `agent:implementing`, `agent:pr-review`, `agent:fixing`, `agent:merge-ready`, `agent:done`, `agent:blocked` 在内的 8 到 9 个 Label。
* **代码实现**：`src/state/labels.ts` 将状态 `PrOpened`, `PrReviewing`, `CiWaiting` 均映射至统一的 `agent:pr-review` Label。这在工程上进行了合理的简并，但使状态机中的细分阶段（如等待 CI）在 GitHub 标签层面失去了独立表示。

### 1.4.2. 幂等键 (Idempotency Key) 格式
* **合同描述**：统一格式为 `run_id:state:head_sha:action`。
* **代码实现**：代码在不同场景下使用了多种变体，例如 `${runId}:transition:${eventType}:${nextState}` 和 `${runId}:planner:plan-comment`。虽然功能上同样能够保障幂等，但未严格遵循合同定义的命名模板。

### 1.4.3. CLI 命令与 Exit Codes
* **设计要求**：设计支持 exit code 0-4 分别对应成功、一般失败、外部依赖错误、状态冲突和配置缺失。
* **代码实现**：`src/cli.ts` 实现了全部 9 个设计命令（`serve`, `reconcile`, `ui` 等），但目前的 CLI 退出码大多仅区分 0（成功）与 1（失败），并未对 2-4 进行细化处理。

## 1.5. 实现领先于文档的项（建议同步更新设计方案/合同）

### 1.5.1. 引入 Triage（分级）角色与 triage 阶段
* **代码实现**：代码中新增了 `triage` 角色（`AgentRole.Triage`），并在 `src/orchestrator/triage.ts` 和 `issue-dispatch.ts` 中实现了分类处理机制，过滤非范畴内（out of scope）的 Issue。
* **文档状态**：主设计方案未提及该角色，JSON Schema (`triage-result.schema.json`) 与 API 设计文档已经对其有所体现。方案应当补齐 Triage 阶段的逻辑流转。

### 1.5.2. UI Server 服务 (`ao ui`) 与静态页面路由
* **代码实现**：`src/ui/server.ts` 提供了一个包含 Dashboard 和 Run/Delivery 列表的 Web UI，并且 `src/state/sqlite-queries.ts` 实现了对应的 WAL 只读模式和 PRAGMA 查询机制。
* **文档状态**：虽然在 API 设计（08-local-ui-api）中对其进行了描述，但主设计方案中并未提及此功能的架构规划。

### 1.5.3. 自动修复与状态修复 (State Repair)
* **代码实现**：`src/reconciliation/state-repair.ts` 实现了 `repairStateFromArtifacts`，允许在本地 SQLite 数据损坏时通过 GitHub 上的 Marker 重新回填还原状态，生成 `reconciliation.repaired` 事件。
* **文档状态**：合同及设计方案未对其进行约束。

## 1.6. 设计漂移与不一致点（高优先级，需核对与修正）

### 1.6.1. Webhook 服务端点路径不一致 (Major)
* **设计文档** (`04-management-apis.md` L7)：描述的 Webhook 接收端点为 `POST /webhooks/github`。
* **代码实现** (`src/cli.ts` L1089, L470)：实际的路由判定和 live-smoke 客户端发送的路径为 `POST /webhook`（单数且不带 github 路径）。

### 1.6.2. 异常 HTTP 响应状态码不一致 (Major)
* **设计文档** (`04-management-apis.md` L22, L24)：规定 Webhook 签名不匹配返回 `401`，重复交付（duplicate delivery）返回 `202`。
* **代码实现** (`src/cli.ts` L1288, L1199)：签名校验失败时返回 `400`（异常捕获区统一抛出），重复交付时返回 `200`（而非 202）。

### 1.6.3. PR Review Webhook 事件缺失与逻辑改变 (Major)
* **设计文档** (`github-native-agent-orchestrator-自动处理-issue-方案.md` §9.1 L729)：规定应解析 `pull_request_review.submitted` 事件以获取 APPROVED 或 CHANGES_REQUESTED 状态。
* **代码实现** (`src/webhooks/domain-event.ts`)：代码中**并未**实现 `pull_request_review` 的事件解析。相反，PR review 流程目前在 `runtime-lifecycle.ts` 中是以同步的 inline agent 执行或由 issue 上的评论 mention 触发 dispatcher 形式运转，无法异步接收纯 Webhook 驱动的外部 PR review 结论。

### 1.6.4. 仓库策略配置文件格式漂移 (Major)
* **设计文档** (§5.2, L346)：定义的配置文件为 YAML 格式的 `.github/agent-orchestrator.yml`。
* **代码实现** (`src/policy/repo-policy-loader.ts` L33)：实际的配置文件被硬编码为 JSON 格式的 `.github/agent-orchestrator.json`，在代码中直接使用 `JSON.parse` 进行解析。

### 1.6.5. Marker Verdict 枚举不匹配 (Major)
* **代码实现** (`src/orchestrator/plan-comments.ts` L81)：渲染的 fix 阶段 verdict 标记为 `FIX_READY`。
* **JSON Schema** (`agent-marker.schema.json` L23-31)：该 verdict 枚举字段并未包含 `FIX_READY`。这会导致生成包含 `FIX_READY` 的 Marker 时无法通过自身的 JSON 校验。

### 1.6.6. PR Review 模板与 Blocking Findings 段缺失 (Minor)
* **设计文档** (`github-artifact-templates.md` L143, L149)：PR 评审报告标题为 `## Agent PR Review`，且应包含一个 `## Blocking Findings` 的段落。
* **代码实现** (`src/orchestrator/plan-comments.ts` L89-104)：标题渲染为 `## PR Review`，并且缺失了 `## Blocking Findings` 段，仅包含摘要与 verdict。

### 1.6.7. DomainEvent.source 字段类型收窄 (Major)
* **JSON Schema** (`domain-event.schema.json` L52)：source 枚举值为 `["webhook", "reconciliation", "manual"]`。
* **代码实现** (`src/webhooks/domain-event.ts` L29)：TypeScript 接口中 `source` 被强类型限制为单一值 `"webhook"`，导致无法在代码类型层面表示来自于 reconciliation 或 manual 触发的域事件。

## 1.7. 建议与下一步行动

1. **统一 Webhook 路径与状态码**：
   * 修正 `cli.ts` 使 Webhook 接收端点符合 `/webhooks/github` 设计，或更新 `04-management-apis.md` 协议文档以对齐实际实现的 `/webhook`。
   * 修正 `cli.ts` 异常捕获，在签名校验失败时返回 `401`，在交付重复时返回 `202`。
2. **规范化 `AgentRole` 与 `merge_agent`**：
   * 将 `merge_agent` 明确为 marker role，不纳入 `AgentRole` 枚举，解耦 LLM 角色定义与内置确定性（deterministic）代码执行。
   * 在设计文档中补充 `triage` 角色和相应的分发阶段设计。
3. **配置文件格式规范化**：
   * 鉴于目前的 JSON 实现更为便捷，建议将方案文档中的 `.github/agent-orchestrator.yml` 更新为 `.github/agent-orchestrator.json` 并更改相应说明。
4. **修复 Schema 与 Verdict 不匹配问题**：
   * 在 `agent-marker.schema.json` 的 verdict 枚举中增加 `FIX_READY`。
   * 拓宽 `src/webhooks/domain-event.ts` 中的 `DomainEvent.source` 强类型定义，允许 `"reconciliation" | "manual"`。
5. **在方案文档中明确 Webhook 处理的裁剪**：
   * 在主方案文档 §9 中明确说明：`pull_request.opened`、`pull_request_review.submitted`、以及 `status` 事件在 MVP 阶段不进行直接 Webhook 事件映射，分别通过评论 trigger、reconciliation 轮询或 check_run 进行合理降维和等价替代。

## 1.9. 结论

系统实现具有较高的质量和整体设计一致性，核心的状态机与核心接口完全落地。目前表现出的漂移大多属于 MVP 开发过程中的**工程裁剪**（如 Webhook 事件的规范化降维）与**配置格式简化**（YAML 转 JSON）。应当尽快将本报告中的高优先级事项（特别是配置文件格式、端点路由、Verdict 校验枚举和 DomainEvent TS 类型定义）在设计与代码层进行双向对齐，以避免长期的技术债。
