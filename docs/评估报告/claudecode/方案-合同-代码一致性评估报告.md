| 维度 | 评价 | 备注 |

## 1.4. 中度一致的项（结构正确，存在小差异）

### 1.4.1. CLI flag 子集

- `ao live-smoke` 在合同里要求支持 `--delivery <id>`，`cli.ts` 实现里同样支持。但合同中说明的 `--body`、`--title` 也都在代码里有相应支持。
- `ao init-config` 的 `--agent-command` 合同里写的是"role agent command"，实际实现里是按角色注入的（`buildProcessAgents(config)`），符合"全局默认 + 仓库覆盖"的预期。

### 1.4.2. 域事件枚举扩展

- 方案 §9.1 列出的必处理事件：`issues.opened / labeled / unlabeled / issue_comment.created / pull_request.opened/synchronize/reopened / pull_request_review.submitted / check_run.completed / status / workflow_run.completed`。
- 实现 `src/webhooks/domain-event.ts:3-13` 的 `DomainEventType` 仅 9 个枚举（`IssueAutopilotRequested`/`IssueCommentDispatchRequested`/`ControlPause`/`ControlResume`/`ControlNoMerge`/`PullRequestSynchronized`/`ChecksSucceeded`/`ChecksFailed`/`ChecksPending`），覆盖了方案中"labeled → autopilot"、"comment mention"、"control labels"、"PR synchronize"、"check_run/workflow_run 收尾"。
- **没有产出独立 `DomainEventType`**：
  - `pull_request.opened`（新建 PR 后用 `PullRequestSynchronized` 还是新增事件？）
  - `pull_request_review.submitted`（PR review webhook 实际是构建 `IssueCommentDispatchRequested`，见 `src/orchestrator/runtime-lifecycle.ts:105-298`——但 `triage.ts:122-127` 的 `PrOpened/PrReviewing → pr_reviewing` 走的是 dispatcher 路径）
  - `issues.unlabeled`（不是 `agent:pause` 时不会触发 `ControlResume`）
  - `status`（GitHub legacy commit status）——实现里只解析 `check_run` / `workflow_run`，不解析 `status` 事件
  - `check_run.completed` 与 `workflow_run.completed` 都通过 `checkRunEventType()` 归一化为 `ChecksSucceeded`/`ChecksFailed`/`ChecksPending`，符合"workflow_run 只能作为补充信号"的合同。
- 评价：实现做的是规范化与降维（用 dispatcher + 三类 check 事件覆盖方案中的必处理事件），是合理的工程取舍，但需要在方案/合同里同步写明"PR review webhook 实际由 issue_comment mention 触发 dispatcher"以及"legacy status 事件不单独处理"的策略。

### 1.4.3. `agent:done` 标签

- 方案 §8.2 列出 `agent:done`。
- 实现 `src/state/labels.ts:17-18` 把它映射到 `Merged` 与 `IssueClosed` 两个状态——与方案一致，但合同里没有显式声明 `agent:done`，应补充。

### 1.4.4. `risk:medium` 风险等级

- 方案 §8.4 列出 `risk:low/medium/high`。
- 合同里 `ReviewerVerdict` 枚举和 `MergeDecision` 风险都接受 `low|medium|high`。
- 实现 `src/orchestrator/merge-gate.ts:30-40` 入参 `risk: "low" | "medium" | "high"`，`allowedRisks: readonly (...)[]`——支持 `medium`，与方案一致。
- 但合同 `task-state-contracts.md` 的 autopilot 启动条件仅讨论 `agent:pause` 和 `needs-human`，没有明确如何根据 `risk:medium` 决策；实现里只把它纳入 `risk_allowed` 检查。这是文档缺失而非实现偏离。

### 1.4.5. 权限字段

- 方案 §13.1 列出"Administration: read"为可选权限。
- 实现 `src/contracts/validation.ts:46` 的 `checks.source` 枚举包含 `"github_merge_gate" | "policy_required_names" | "branch_protection_read"`，与方案 §5.2 描述一致。
- 合同里只在 `06-error-codes-and-permission-actions.md` 的 Permission Actions 表中显式声明权限，方案 §13.1 的细节没有完整反映到合同。

### 1.4.6. 任务信封

- 方案 §7.2 给出的任务信封示例只包含 `role/issue/plan/workspace/policy/expected_outputs`，没有 `repo.default_branch`、`pr`、`dispatch` 等。
- 实现 `src/agents/adapter.ts:33-88` 与 schema 都更完整（加了 `repo.default_branch`、`pr`、`dispatch`、`policy.allow_secrets`）。这是方案示例的简化，不是矛盾；建议在方案里补全。

## 1.6. 设计漂移 / 偏离点（高优先级，需要决定对齐方向）

### 1.6.1. `merge_agent` 在 `AgentRole` 里缺失但合同层又存在

- 方案 §4.4 / §11.5 / §6.1 / §6.2 明确 `merge_agent` 是一个独立逻辑角色，由"builtin deterministic"承载。
- 实现 `src/agents/adapter.ts:3-9` 的 `AgentRole` 枚举**不包含** `merge_agent`；`LocalConfig` 里的 `agents.merge_agent` 字段只是配置占位（`{ adapter: "builtin"; mode: "deterministic" }`），并不驱动 `ProcessAgentAdapter`。
- 实际 merge 行为由确定性代码（`src/orchestrator/runtime-lifecycle.ts:1347-1436` 的 `finishMergeAndCloseout`）执行。
- 在 `src/github/markers.ts:11` 和 `src/reconciliation/github-artifacts.ts:257` 里又把 `merge_agent` 当作"marker role"使用——所以 `merge_agent` **不是 AgentRole，而是 marker role**。
- **不一致点**：
  - 合同 `task-state-contracts.md:95` "final summary marker with role `merge_agent` and verdict `MERGED`" 与实现一致；
  - 合同 `agent-marker.schema.json`（如果存在 role 枚举）需要确认是否包含 `merge_agent`；
  - 方案 §6.2 / §11.5 与 `LocalConfig` 的 `agents.merge_agent` 字段使用了 LLM-角色语义，与"marker role"语义混在一起。
- 建议：在方案/合同里把"merge_agent"明确为 marker role（不参与 AgentRole / AgentAdapter），与实现保持一致。

### 1.6.2. 状态 ↔ label 映射与方案 §8.2 不完全一致

- 方案 §8.2 列出 8 个状态 label：`agent:planning` / `agent:plan-review` / `agent:implementing` / `agent:pr-review` / `agent:fixing` / `agent:merge-ready` / `agent:done` / `agent:blocked`。
- 实现 `src/state/labels.ts:8-21` 把 `PrOpened`、`PrReviewing`、`CiWaiting` 三个内部状态都映射到同一个 `agent:pr-review`，与方案一致（8 个状态 label）。
- 但合同 `task-state-contracts.md` 列出"State labels are mutually exclusive"后给出 9 个 label：`agent:planning` / `agent:plan-review` / `agent:implementing` / `agent:pr-review` / `agent:fixing` / `agent:merge-ready` / `agent:done` / `agent:blocked`——实现也是这 8 个（`agent:done` 覆盖 `merged/issue_closed`），不构成漂移。
- 唯一需要确认：合同里 `agent:pr-review` 是否要拆成 `agent:pr-opened` + `agent:pr-reviewing` + `agent:ci-waiting`。当前实现合并为一项，方案也是合并的——可视为一致。

### 1.6.3. 方案 §6 角色配置 vs `LocalConfig`

- 方案 §6.2 的 agent 配置示例使用 `adapter: codex/claude/builtin`、`command`、`args`、`mode: read_only/write_worktree/deterministic`。
- 实现 `src/contracts/validation.ts:94-100` 的 `AgentConfig` 实际形状是：
  - `adapter: "codex" | "claude" | "custom"`
  - `command: string`
  - `args: readonly string[]`
  - `mode: "read_only" | "write_worktree"`
  - `network?: "deny" | "allow" | "restricted"`
- 差异：
  - 实现不支持 `adapter: "builtin"`（除 `merge_agent` 字段单独定义），方案示例里的"内置"是通过 `merge_agent` 占位。
  - 实现的 adapter 多了 `custom` 取值，方案没写。
  - 方案 `mode` 多了 `deterministic` 取值；实际只在 `merge_agent` 占位字段里出现。
- 建议：在方案 §6.2 / §6.3 / §6.4 中移除 `mode: deterministic` 与 `adapter: builtin`，统一用 `adapter: codex/claude/custom` + `mode: read_only/write_worktree` 表达。

### 1.6.4. 方案 §9.2 触发规则 vs `DomainEvent` 字段

- 方案 §9.2 描述"PR body 包含 `Closes #<issue>` 才进入 review"等条件。
- 实现 `src/webhooks/domain-event.ts:100-112, 224-241` 只对 `pull_request` 事件处理 `synchronize` action；`pull_request.opened` 走的是 `IssueCommentDispatchRequested` 路径（PR 评论 @agent 时）——即由"评论触发"，不是由"PR 打开触发"。
- 因此"PR 打开后自动进入 review"在实现里**不会自动发生**，需要：
  - 要么 PR 的 issue 评论里 @agent 触发；
  - 要么 reconciliation 在扫描时进入 review。
- 建议：在方案 §9.2 / §10 状态失效规则中明确："PR opened 不直接驱动状态机；新 PR 的状态推进由 Issue 评论 mention 触发，或由 reconciliation 触发"。

### 1.6.5. 方案 §11.3 Implementer 约束中的"测试命令"

- 方案 §11.3：Implementer 输入"测试命令"。
- 实现 `src/agents/adapter.ts:68-76` 的 `TaskEnvelope.policy.required_tests` 是字符串数组，没有"命令"概念，由 Implementer 自己决定怎么跑。合同 `task-envelope.schema.json` 也只要求 `required_tests: string[]`。
- 评价：方案描述粒度更细，实现粒度更粗；建议在方案里把"测试命令"弱化为"测试名/命令的字符串列表"，与 schema 对齐。

### 1.6.6. 方案 §14.4 "merge_method" 默认值

- 方案 §14.4 "默认 merge 方法：squash" + `policy.merge.default_method` 字段在合同/实现里都允许 `"squash" | "merge" | "rebase"`。
- 一致。

### 1.6.7. 方案 §10 状态机的"PR review 无 verdict / PR review APPROVED on stale head" 处理

- 方案 §9.2 提到 "Reviewer verdict 为 `APPROVED` 时进入 CI gate"，但未明确"reviewer 返回 `null` 或 `verdict` 不在 enum 时"的处理。
- 实现 `src/orchestrator/pr-gate.ts:78-89` 的 `mapPrReviewVerdictToEvent`：verdict 不匹配 `APPROVED/REQUEST_CHANGES` 时返回 `AgentPrReviewBlocked`；head_sha 不匹配时返回 `undefined`，由 `runtime-lifecycle.ts:424-428` 抛 `STALE_HEAD_SHA`。
- 合同里 `STALE_HEAD_SHA` 的"Blocks: Maybe"是含糊的。实际实现里"`undefined` 即抛错"是隐式行为，文档需要显式说清。

### 1.6.8. 方案 §14.3 高风险路径 vs 实现

- 方案 §14.3 默认高风险路径包括 `**/*permission*`、`**/*policy*`；合同 `06-error-codes-and-permission-actions.md` 没有列出默认高风险路径。
- 实现 `src/policy/path-policy.ts` 的"高风险"完全是 `policy.paths.high_risk` 字段决定，无内置默认。`runtime-lifecycle.ts` 把 `POLICY_HIGH_RISK_PATH` 错误码映射到 `blockRunForPathPolicy`，并且会抛错、不会自动加 `needs-human` 标签——但方案要求"自动加 `needs-human` + `agent:blocked` + 评论阻断原因"。
- 评价：实现里 `path-policy` 只做"决策 + 抛错"，**没有自动添加 `needs-human` / `agent:blocked` 标签的副作用**。`buildBlockedHandling` 似乎会加 label（`src/orchestrator/workflow-control.ts`），需要再核验。结论：在 high_risk 命中时是否写回 `needs-human` / `agent:blocked` 两个 label 是合同/方案的关键要求，实现目前是抛错为主，label 副作用待二次确认。

## 1.11. 结论

- 实现、合同、API 设计三者高度自洽，状态机、表结构、错误码、CLI、Adapter 接口、merge gate 全部对齐。
- 方案文档（`github-native-agent-orchestrator-自动处理-issue-方案.md`）是较早期设计稿，与实现/合同存在 3 个明显漂移点：缺少 Triage 角色、`merge_agent` 角色语义错位、PR opened 事件未被规范化。
- 优先建议：先在方案/合同里补齐 Triage 与 marker role，再把 `AgentConfig` 的 adapter/mode 集合统一，最后统一 idempotency key 格式与 high_risk 副作用文档。
