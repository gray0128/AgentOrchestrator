## 4. 分文档详细对照

### 4.1 主方案文档 `github-native-agent-orchestrator-自动处理-issue-方案.md`

#### 与代码一致的部分

- GitHub 为事实来源、SQLite 为调度缓存的定位（§3、§4.6）。
- 核心 Agent 角色：planner / plan_reviewer / implementer / pr_reviewer / merge_agent（merge 为确定性代码）。
- 端到端流程：Issue → 方案 → 方案审核 → 实现 → PR → PR 审核 → CI → 合并 → 关 Issue。
- Label 驱动、`agent:autopilot` 入口、状态 label 互斥、控制 label、风险 label。
- 状态机主路径（§10）与 `state-machine.ts` 一致。
- 修复循环最多 3 轮（由 `review.max_fix_rounds` 策略控制）。
- Orchestrator Server + GitHub App 分工（§4.1–4.2）。

#### 与代码/契约不一致或缺失的部分

1. **策略与配置格式**：全文使用 YAML 示例（§5.2、§5.3、§6.2），实现为 JSON Schema 校验的 JSON 文件。
2. **无 Web UI 声明**：与已实现的 `ao ui` 只读运维面矛盾。
3. **无 Triage 阶段**：实现中 autopilot/mention 触发后先跑 Triage，再决定 `next_step`。
4. **Webhook 清单大于实现**：§9.1 列出的 `pull_request_review.submitted`、`status` 等未归一化。
5. **Issue 评论控制命令**：§6.3 `/agent pause` 等未实现。
6. **SQLite 示例 schema 过时**：缺 `idempotent_actions` 表说明中的部分字段、错误列命名不同。
7. **Agent 动态路由**：§6.4 仓库级 `routing.rules` 未实现；本地 `agent_routing` 替代。

## 7. 风险分级

### P0 — 影响生产正确性或安全语义

| 项 | 风险 | 说明 |
| --- | --- | --- |
| `githubMergeable: true` 硬编码 | 中 | 本地 merge gate 可能误判可合并；最终仍依赖 GitHub Merge API，但会浪费轮次或错误进入 merge 尝试 |
| `PROMPT_INJECTION` 未 enforcement | 中 | 契约要求阻断，代码仅注册错误码 |
| 移除 `agent:autopilot` 不停止 run | 中 | 主方案 §10 要求取消/超时，当前忽略 `issues.unlabeled` |

### P1 — 功能缺口，可用 workaround

| 项 | 说明 |
| --- | --- |
| 无 `pull_request_review.submitted` webhook | 依赖 agent 自产 review + lifecycle 内联，外部人工 review 可能需 reconcile/resume |
| `/agent` 评论命令未实现 | 可用 label 替代 |
| `reconcile --apply` 不自动续跑 lifecycle | 需 webhook 或手动 dispatch |
| `branch_protection_read` 未实现 | 依赖 check-runs + Merge API |

### P2 — 文档漂移

| 项 | 说明 |
| --- | --- |
| 主方案 YAML / 无 UI / 无 Triage | 误导新接入者 |
| api-design webhook 路径不一致 | 配置 GitHub App 时易混淆 |
| development-plan「不做 Web UI」 | 与 api-design 冲突 |

## 8. 修复建议（按优先级）

### 8.1 文档（低成本、高收益）

1. **更新主方案文档**：策略文件改为 JSON；补充 Triage、mention 触发、`ao ui`（只读运维）；配置示例改为 `config/local.json`；SQLite 示例与 `data-contracts.md` 对齐。
2. **修正 `04-management-apis.md`**：Webhook 路径改为 `POST /webhook`。
3. **更新 `development-plan/README.md`**：MVP 包含只读本地 UI，非用户控制台。
4. **更新 `contracts/README.md`**：Required Schemas 加入 `triage-result`；`action-proposal` 标为预留或移出 Required。
5. **在 `repo-policy.schema.json` 或契约中注明**：`routing.rules` 为预留，MVP 路由见 `local-config.agent_routing`。

### 8.2 代码（按里程碑）

1. 读取 PR `mergeable` / `mergeable_state` 填入 `evaluateMergeGate`（对齐 Query API 设计）。
2. 实现 `issues.unlabeled`（`agent:autopilot`）→ pause/cancel run。
3. 解析 Issue 评论 `/agent pause|resume|retry|no-merge`（对齐 `04-management-apis.md`）。
4. 归一化 `pull_request_review.submitted`（可选 `status`）为 domain events。
5. Prompt injection 规则最小检测（策略关键字、token 请求等）→ `blocked`。
6. `reconcile --apply` 可选 `--execute` 在 claim 后对 eligible run 调 `runIssueLifecycleFromStep`。

## 9. 结论

- **契约层（`docs/contracts/`）与 API 设计（`docs/api-design/`）是当前最可信的实现真源**；与 `src/` 核心编排逻辑高度一致，215 项测试支撑主要不变量。
- **主方案文档是主要不一致来源**：格式、配置、UI、Triage、路由模型均已演进，需同步修订以免接入与评审偏差。
- **剩余差距集中在「事件覆盖、人工控制面、merge 预检、安全 enforcement」**，多数有明确契约条目，适合拆为独立 GitHub issue 逐项闭合。
