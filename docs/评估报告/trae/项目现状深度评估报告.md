# AgentOrchestrator 项目现状深度评估报告

## 1. 评估范围与方法

本次评估基于仓库当前工作区内容进行，不参考也不继承已有 `docs/review-report.md` 的结论。

评估输入包括：

- 顶层方案文档 `github-native-agent-orchestrator-自动处理-issue-方案.md`
- 进度文档 `docs/progress/`
- 契约文档 `docs/contracts/`
- API 设计文档 `docs/api-design/`
- 实现代码 `src/`
- 测试代码 `test/`
- 使用文档 `README.md`
- 当前仓库实际校验结果

本次评估采用三类证据交叉验证：

1. 文档证据：确认项目目标、边界、阶段与约束。
2. 代码证据：确认模块落地情况、能力边界与架构组织方式。
3. 运行证据：执行 `npm test` 与 `npm run check`，核对当前仓库可验证状态。

## 2. 执行摘要

总体判断：项目已经完成 MVP 到可运行产品化形态的主体建设，不再处于“设计中”或“半成品”阶段，而是处于“核心能力已落地，下一阶段应转向审计、加固、运维化和体验优化”的状态。

核心结论如下：

- 项目目标明确，且方案、契约、API 设计、开发计划、进度台账之间整体一致。
- 代码结构清晰，职责边界较明确，属于典型的“Webhook 驱动 + 状态机编排 + GitHub API 回写 + 本地 SQLite 持久化”架构。
- 当前仓库实现已经覆盖 CLI、Webhook 服务、状态机、GitHub 适配层、Agent 适配层、策略校验、状态修复、本地只读 Web UI 等关键能力。
- 测试成熟度较高，当前执行 `npm test` 结果为 158 个测试全部通过，说明主干能力具备较好的回归保护。
- 文档化和验收记录做得比较完整，`docs/progress/test-acceptance-log.md` 与 `docs/progress/contract-checklist.md` 可作为较强的“工程完成证据”。
- 当前最主要的问题不是核心功能缺失，而是“加固阶段尚未任务化”，以及仍存在少量工程化短板，例如覆盖率指标缺失、UI 缺少浏览器级验证、个别核心流程模块缺少更直接的单元测试。
- 当前 `npm run check` 无法全绿的直接原因不是源码失败，而是工作区内现有文件 `docs/review-report.md` 未通过格式检查；该问题与本次新增报告无直接关联，但会影响仓库整体门禁表现。

一句话结论：这是一个已经具备真实运行能力的 GitHub-native 自动化编排器原型产品，工程成熟度已超过 MVP 起步阶段，但距离“长期稳定运营级”仍需要一轮面向质量和可运维性的系统加固。

## 3. 当前项目定位

从方案文档、README 和当前代码实现看，项目定位非常稳定：

- 以 GitHub 为唯一用户可见事实源。
- 以 GitHub App + Webhook 作为外部触发机制。
- 以本地 Orchestrator 作为状态推进与副作用执行中枢。
- 以 SQLite 作为本地状态、幂等、租约、审计轨迹的存储层。
- 以本地 coding agent 进程作为规划、实现、审核等角色执行体。

当前明确不在范围内的能力也保持一致：

- 非 GitHub 仓库场景不支持。
- 多仓库事务不支持。
- 托管式 Web UI 不支持。
- 不绕过 GitHub branch protection、required checks、required reviews 和 rulesets。

这说明项目在“做什么”和“不做什么”上边界相对清晰，没有明显的方向漂移。

## 4. 代码与工程资产现状

### 4.1 仓库资产规模

基于当前工作区统计：

- `src/` 共 54 个文件
- `test/` 共 41 个文件
- `docs/` 共 43 个文件

这说明项目已经形成较完整的“实现 + 测试 + 文档”三层资产结构，而不是单纯靠 README 驱动的原型仓库。

### 4.2 模块结构

当前源码主要分为以下模块：

- `src/orchestrator/`：编排核心，负责生命周期推进、PR gate、merge gate、收尾与控制流。
- `src/webhooks/`：Webhook 验签、去重、事件归一化、评论触发分发。
- `src/state/`：SQLite 状态表、查询、CAS 状态迁移、标签同步。
- `src/github/`：GitHub App 鉴权、REST API 适配、marker 与请求幂等辅助。
- `src/agents/`：Agent 接口、进程型 adapter、路由 adapter、fake adapter。
- `src/policy/`：actor 控制、路径策略、repo policy 加载。
- `src/reconciliation/`：本地状态修复、GitHub artifact 回放、dry-run 核对。
- `src/ui/`：本地只读 Web UI 和本地 JSON API。
- `src/workspace/`：工作区路径与分支规划。
- `src/contracts/`：配置和 schema 校验逻辑。
- `src/security/`：敏感信息脱敏。

模块边界较清晰，没有出现“大量逻辑全部堆进 CLI 或单一 service 文件”的失控迹象。

### 4.3 入口与运行形态

当前产品化入口已经比较完整：

- `ao init-config`
- `ao doctor`
- `ao validate`
- `ao live-check`
- `ao serve`
- `ao live-smoke`
- `ao reconcile`
- `ao inspect-run`
- `ao ui`

这意味着项目已经从“开发者内部脚本”进化为“有明确 operator 操作面”的本地工具产品。

### 4.4 技术栈特征

当前技术栈选择偏克制：

- 语言：TypeScript
- 运行时：Node.js `>=26.0.0`
- HTTP：Node 原生能力
- 数据层：SQLite
- 测试：Node 内置 test runner
- 打包：Node SEA + esbuild

优点：

- 依赖轻，技术面集中。
- 可执行边界清晰，部署和排障相对简单。
- 适合内部小项目“效率优先、稳定优先、交付优先”的目标。

代价：

- 对 Node 26 的依赖较强，环境要求偏前沿。
- 自研边界较多，后续需要靠测试和文档维持一致性。

## 5. 架构实现成熟度评估

### 5.1 主链路完整度

从当前实现看，主链路已经闭环：

1. 接收 GitHub Webhook。
2. 校验签名与 delivery 去重。
3. 归一化为内部 domain event。
4. 基于 Issue/PR/check/review 事件推进状态机。
5. 通过不同角色的 agent 生成方案、实现和审核结果。
6. 通过 GitHub API 执行 comment、label、branch、commit、PR、review、merge、close 等动作。
7. 将状态、转换、幂等动作、delivery 记录回 SQLite。
8. 提供 reconcile 与 UI 查询能力，支持恢复与可视化。

这是一个结构完整的 orchestrator，而不是局部功能的集合。

### 5.2 状态机成熟度

状态集合和迁移规则已经系统化，核心状态包括：

- `new`
- `planning`
- `plan_reviewing`
- `implementing`
- `pr_opened`
- `pr_reviewing`
- `ci_waiting`
- `fixing`
- `merge_ready`
- `merged`
- `issue_closed`
- `paused`
- `blocked`
- `failed`

状态机成熟度较高的原因：

- 有明确的状态词汇表。
- 有显式的允许迁移表。
- 有 head SHA 失效规则。
- 有 pause、blocked、needs-human 等控制语义。
- 有 reconciliation 与 stale-head 防护。

这类设计对于 GitHub 事件驱动系统至关重要，当前实现属于项目的强项。

### 5.3 GitHub 集成成熟度

GitHub 集成不是停留在 mock 阶段，而是已经覆盖：

- GitHub App 鉴权
- installation token 获取与缓存
- issue comment
- label 写入
- branch 创建
- commit 写入
- PR 创建或复用
- review 写入
- checks/status 读取
- merge 执行
- branch 删除
- issue close

并且实现中强调：

- 写操作必须带幂等语义
- merge 必须绑定当前 `head_sha`
- agent 不直接持有 GitHub token

这说明项目没有把安全和一致性问题留到以后再补，而是较早进入了实现主干。

### 5.4 Agent 编排成熟度

Agent 适配层的设计相对稳健：

- 提供统一任务信封和输出 schema。
- 支持 fake adapter，便于测试。
- 支持 process adapter，便于接真实命令。
- 支持 routing adapter，支持多候选 agent 与 reviewer 池。

这让系统从一开始就不是“绑定单一模型 CLI 的脚本”，而是一个具备一定可替换性的编排框架。

### 5.5 可观测性与恢复能力

项目在内部可观测性上已经做了不少工作：

- delivery 记录
- workflow run 状态
- state transition 审计
- idempotent action 记录
- inspect-run 命令
- reconcile dry-run
- 本地 UI

这对“事件驱动 + 外部副作用”系统非常重要。当前已经具备排障和恢复的基础设施，这一点明显优于多数早期原型。

### 7.2 测试覆盖优势

当前测试优势比较明显：

这不是“没有测试”，而是“已具备回归保护，但还没有做到质量可视化和更细颗粒度的防线”。

### 8.2 契约先行且落地

Schema、状态、数据、安全、GitHub artifact 均有文档和实现映射，这让系统比常见内部自动化脚本更可维护。

### 8.3 工程纪律较好

存在开发计划、任务清单、契约清单、验收日志、版本发布记录，说明项目不是一次性实验，而是带治理痕迹的持续建设。

### 8.4 质量基础较强

158 个测试全通过，且测试覆盖的不仅是纯函数，还包括 CLI、HTTP、SQLite、GitHub adapter、reconcile、E2E smoke。

### 8.5 可恢复性设计较成熟

lease、CAS、idempotency、head invalidation、artifact reconciliation 这些机制，说明系统已经充分考虑实际 webhook 运行中的重复、乱序和中断问题。

## 9. 当前主要风险与短板

### 9.1 仓库门禁存在现实阻塞点

当前 `npm run check` 被 `docs/review-report.md` 的格式检查阻塞。即使主代码和测试是好的，这也会影响“仓库是否完全可提交/可发布”的判断。

影响：

- 会让 CI 或本地标准检查不全绿。
- 会掩盖真正的代码问题，因为失败先停在格式检查。

### 9.2 Hardening 阶段没有继续任务化

当前文档明确说明项目进入 audit/hardening 阶段，但没有看到后续任务拆解。这会导致：

- 下一阶段缺少统一优先级。
- 很难判断“加固是否完成”。
- 质量工作容易碎片化、机会化。

### 9.3 质量指标偏结果导向，缺少量化仪表

当前有很多“是否通过”的验证，但缺少一些工程化指标：

- 覆盖率阈值
- 关键路径回归矩阵
- UI 自动化验证指标
- 性能或稳定性基线

这会影响项目从“可用”走向“可持续维护”。

### 9.4 Node 26 运行前提提高了环境门槛

当前明确要求 Node.js `>=26.0.0`，对内部项目并非不能接受，但会带来：

- 本地环境准备门槛提升
- SEA 构建依赖官方 Node 发行版
- 一部分运行/构建问题更可能出现在环境层而不是业务层

### 9.5 UI 仍偏运维辅助工具，而非成熟控制台

当前本地 Web UI 已实现 MVP，但仍主要服务于本地观察和调试：

- 只读
- 本地绑定
- 无浏览器自动化测试
- 缺少更强筛选、比对、聚合分析能力

这符合当前范围，但如果未来 operator 使用频率上升，UI 会较快成为后续投入点。

## 11. 建议的下一阶段重点

建议不要再继续无目标扩功能，而是优先做以下几类工作。

### 11.1 先解决仓库标准检查阻塞

优先处理 `docs/review-report.md` 的格式问题，让 `npm run check` 在当前工作区可完整通过。

### 11.2 把 hardening 明确拆成任务

建议在 `docs/progress/tasks.md` 中新增一轮加固任务，至少覆盖：

- 覆盖率统计与阈值
- 核心编排模块定点测试补强
- UI 浏览器级冒烟验证
- 安全脱敏边界测试补强
- release/check 流程稳定化

### 11.3 建立质量可视化指标

建议引入最小化但稳定的质量指标，而不是过度复杂化：

- `npm test` 结果继续保留
- 增加覆盖率输出
- 对关键命令建立 smoke matrix
- 对 live 模式建立 operator readiness checklist

### 11.4 明确“可运营版本”的完成标准

当前更像“已完成的本地产品原型”。如果下一步目标是“稳定运营”，建议补充一版完成定义，例如：

- 标准门禁全绿
- 加固任务全部完成
- live 模式演练完成
- 关键故障恢复路径有文档有演练
- UI/CLI 的 operator 文档达可交接水平

## 12. 最终结论

AgentOrchestrator 当前并不是“需要从头补课”的项目，而是一个已经完成主要架构与主流程落地、具备较强契约意识和测试基础的内部工程系统。

它当前最重要的事实不是“还有哪些核心功能没做完”，而是：

- 核心功能基本已完成。
- 文档和实现整体一致。
- 测试主干可信。
- 进入下一阶段后，主要矛盾已经从“能不能做出来”转移为“如何稳定、可验证、可维护地持续运行”。

如果只用一句话描述现状：

**这是一个已经具备真实使用潜力的 GitHub-native 自动化编排器，当前最值得投入的方向不是继续扩主功能，而是把加固、验证和运维化做成正式工程资产。**
