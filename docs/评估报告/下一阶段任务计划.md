# AgentOrchestrator 下一阶段任务计划

生成时间：2026-06-26

输入来源：

- 2026-06-25/2026-06-26 期间由 Codex、Reasonix、Trae 生成的三份项目评估报告。
- 三份原始评估报告已整合进本文档；后续不再保留为独立迭代依据。

本计划用于把三份评估报告收敛为后续 GitHub milestone / issue / PR 工作项。后续迭代控制以 GitHub 为准：任务状态、优先级、讨论、验收和合并记录应落在 GitHub milestones、issues 和 PR 上；本地 `docs/progress/*` 不再作为迭代控制面。设计文档、契约文档、README 和运行手册仍随代码 PR 正常更新。

## 1. 综合判断

三份评估报告对项目现状判断基本一致：

- 当前项目已经完成可运行 MVP：CLI、webhook server、SQLite 状态、GitHub App auth、REST adapter、agent process adapter、mock e2e、local UI、release workflow 均已存在。
- 项目具备较好的合同层、schema、状态机、幂等、head sha 约束和测试基础。
- 下一阶段主要矛盾不再是“能不能跑通 happy path”，而是“能否长期、安全、可恢复、可验证地处理真实 GitHub Issue”。
- 后续不宜继续无目标扩功能，应优先补齐主生命周期安全约束、恢复证据、失败修复循环、质量指标、安装入口和运维化能力。

下一阶段建议命名为：

```text
Post-MVP Hardening and Operations
```

建议拆为 5 个 GitHub milestones：

| Milestone | 目标 | 优先级 |
| --- | --- | --- |
| M8 - Runtime Safety Hardening | 让主生命周期不再信任 agent 声明，强制使用实际 diff、workspace、policy、错误码和持久化去重 | P0 |
| M9 - Recovery And Fix Loop | 补齐真实 artifact 恢复、CI/review 修复循环和写成功后崩溃恢复 | P1 |
| M10 - Quality Gates And Verification | 建立覆盖率、失败路径 e2e、UI 浏览器验证和 release/CLI smoke matrix | P1 |
| M11 - Operator Readiness | 修复安装入口、策略示例、生产边界说明、live 演练手册和可运营标准 | P2 |
| M12 - Maintainability Cleanup | 拆分大文件、收敛 public API、清理命名和重复逻辑 | P2 |

## 2. 执行原则

后续每个 issue 应遵循以下规则：

- issue 是迭代控制单元，PR 是验收单元。
- 每个 issue 必须能独立验证，不把多个无关风险塞进一个 PR。
- 涉及状态机、schema、GitHub adapter、安全语义、policy、持久化模型、CLI 行为时，先更新对应设计或契约文档，再改实现。
- 不再用 `docs/progress/tasks.md` 标记任务推进状态；可以在 PR 中更新合同、API、README、运行手册和决策记录。
- 每个 PR 至少运行 `npm run check`；若因环境或外部凭据无法完整验证，PR 描述必须写明替代验证和残余风险。
- live GitHub 行为必须通过 fake/mock 单测覆盖确定性逻辑，通过 operator smoke 覆盖真实边界。

## 3. Milestone M8 - Runtime Safety Hardening

目标：主生命周期必须以真实工作区、真实 diff、策略校验、明确错误码和持久化去重为准，不能依赖 agent 自报或假数据兜底。

### Issue M8-01：将 workspace manager 接入 full lifecycle

建议标题：`M8: wire workspace manager into runtime lifecycle`

背景：

- Codex 报告指出 `src/workspace/manager.ts` 未进入 `runtime-lifecycle` 主路径。
- 当前 implementer 阶段主要使用 agent 输出的 branch / changed_files，工作区隔离和实际 diff 取证不足。

范围：

- 在 implementer 阶段通过 workspace manager 准备受控工作区。
- 统一 branch 命名、workspace path 校验和 diff collection。
- full lifecycle 后续提交、PR body、summary 使用实际工作区证据。

不在范围：

- 不引入多 worker 调度。
- 不重写 agent adapter 协议。

验收标准：

- implementer 不能绕过受控 workspace 写入。
- 工作区路径逃逸会失败并进入明确错误。
- e2e smoke 仍通过。
- 新增测试覆盖工作区逃逸、实际 diff 为空、实际 diff 与 agent 输出不一致。

建议标签：`type:hardening`、`area:runtime`、`area:workspace`、`priority:p0`

需要更新文档：

- `docs/api-design/03-internal-module-interfaces.md`
- `docs/contracts/security-and-audit-contracts.md`
- README 运行边界说明

### Issue M8-02：在 full lifecycle 强制执行实际 diff path policy

建议标题：`M8: enforce path policy against actual git diff before GitHub writes`

背景：

- Codex 报告指出 `evaluatePathPolicy` 当前主要被测试引用，未强制接入 full lifecycle。
- 合同要求实际 changed files 来自 git，而不是 agent 输出。

范围：

- 从实际 git diff 收集 changed files。
- 调用 `evaluatePathPolicy`。
- deny / high-risk / outside allow 自动进入 `blocked`。
- blocked comment 写明文件证据和策略原因。
- 禁止在 policy 未通过时创建 commit / PR / merge。

验收标准：

- deny path 不会被提交。
- high-risk path 自动 blocked。
- outside allow path 自动 blocked。
- full lifecycle 测试证明 policy fail 时没有 GitHub write 副作用。

建议标签：`type:security`、`area:policy`、`area:runtime`、`priority:p0`

依赖：M8-01

需要更新文档：

- `docs/contracts/security-and-audit-contracts.md`
- `docs/contracts/task-state-contracts.md`
- `docs/api-design/06-error-codes-and-permission-actions.md`

### Issue M8-03：移除 fake content 和 fake base sha 兜底

建议标题：`M8: fail explicitly instead of using fake file content or base sha`

背景：

- Reasonix 报告指出 `readChangedFile` 读取失败时返回 `"automation\n"`。
- Reasonix 报告指出 `base_sha ?? "base-sha"` 会产生不明确错误。

范围：

- 文件读取失败必须返回注册错误码。
- 缺少 base sha 必须 fail fast，不生成假 SHA。
- 错误应写入 run last error，并保持可诊断。

验收标准：

- 文件缺失不会生成假 commit 内容。
- base sha 缺失不会调用 GitHub createBranch / commit。
- CLI 和 webhook 响应中错误被脱敏且包含注册错误码。

建议标签：`type:bug`、`area:runtime`、`priority:p0`

需要更新文档：

- `docs/api-design/06-error-codes-and-permission-actions.md`
- `docs/contracts/data-contracts.md`

### Issue M8-04：统一 runtime lifecycle 错误为 OrchestratorError

建议标题：`M8: replace generic lifecycle errors with registered OrchestratorError codes`

背景：

- Reasonix 报告指出 `runtime-lifecycle.ts` 关键路径大量使用泛型 `throw new Error()`。
- 当前关键失败不利于上层分类、重试和 blocked/failed 判断。

范围：

- 为 checks failed、merge gate blocked、agent failed、state conflict、missing artifact、policy denied 等路径使用注册错误码。
- 必要时扩展错误码 registry。
- CLI、webhook、state last error 使用统一 code/message。

验收标准：

- `runtime-lifecycle.ts` 关键业务失败不再使用裸 `Error`。
- 测试覆盖每类错误码映射。
- `npm run check` 通过。

建议标签：`type:hardening`、`area:error-handling`、`priority:p0`

需要更新文档：

- `docs/api-design/06-error-codes-and-permission-actions.md`

### Issue M8-05：将 webhook delivery 去重切换为 SQLite 持久化

建议标题：`M8: persist webhook delivery dedupe records in SQLite`

背景：

- Reasonix 报告指出当前 `startServeRuntime` 使用内存 delivery dedupe，服务重启后丢失。
- SQLite 已有 `deliveries` 数据模型，应作为服务级 dedupe 来源。

范围：

- 新增 SQLite-backed DeliveryStore 或将现有 delivery deduper 接入 state store。
- `POST /webhook` 使用持久化 delivery 去重。
- UI deliveries 页面继续使用同一数据源。
- 重启后重复 delivery 不重复推进状态。

验收标准：

- 同一个 delivery id 在进程重启后仍被识别为 duplicate。
- duplicate 不重复写 GitHub comment / PR / review。
- UI deliveries 能看到 received / processed / ignored / failed 状态。

建议标签：`type:hardening`、`area:webhook`、`area:state`、`priority:p0`

需要更新文档：

- `docs/contracts/data-contracts.md`
- `docs/api-design/08-local-ui-api.md`

### Issue M8-06：改造 agent 环境变量传递为白名单模式

建议标题：`M8: replace agent env blacklist with explicit allowlist`

背景：

- Reasonix 报告指出 `filterAgentEnv` 黑名单可能遗漏敏感变量。
- 当前安全目标是 agent 不接触 GitHub installation token、private key、webhook secret 等。

范围：

- 本地 config 增加可选 safe env allowlist。
- 默认只传递最小必要环境变量。
- 保留兼容模式或 migration note。
- `doctor` 显示会传递的 env key，不显示值。

验收标准：

- 默认不会把任意宿主环境变量传给 agent。
- 测试覆盖 GitHub、NPM、Docker、AWS、custom secret key 不会泄露。
- agent adapter 文档说明 allowlist 行为。

建议标签：`type:security`、`area:agents`、`priority:p0`

需要更新文档：

- `docs/contracts/security-and-audit-contracts.md`
- `docs/contracts/schemas/local-config.schema.json`
- README agent 配置说明

## 4. Milestone M9 - Recovery And Fix Loop

目标：把一次性 happy path 升级为可恢复、可重放、可处理 review/CI 失败的运行主线。

### Issue M9-01：用真实 GitHub artifacts 重建 resume 上下文

建议标题：`M9: rebuild resume context from GitHub artifacts instead of stubs`

背景：

- Codex 报告指出 `runIssueLifecycleFromStep` 使用 stub plan review 和 stub implementation。
- 这会削弱审计和可重放性。

范围：

- 从 issue comments、PR body、reviews、checks、SQLite actions 解析真实 plan、review、implementation、risk、tests、head sha。
- 缺少关键证据时进入 `blocked`，而不是构造 stub。
- 将 artifact recovery 与 reconciliation 逻辑复用。

验收标准：

- resume merge 前使用真实 PR body / review / check 证据。
- 缺 plan marker、缺 implementation marker、缺 current-head review 均 blocked。
- 测试覆盖正常恢复和缺证据恢复失败。

建议标签：`type:hardening`、`area:reconciliation`、`area:runtime`、`priority:p1`

依赖：M8-04

需要更新文档：

- `docs/contracts/github-artifact-templates.md`
- `docs/contracts/task-state-contracts.md`

### Issue M9-02：实现 PR review changes 的完整 fix loop

建议标题：`M9: implement full PR review change-request fix loop`

背景：

- 三份报告都指出当前 fix loop 不完整，更多是状态和决策覆盖。

范围：

- PR reviewer `REQUEST_CHANGES` 后进入 `fixing`。
- 调 implementer 生成 fix result。
- push 新 commit。
- 旧 review/check/merge-ready 结论失效。
- 返回 `pr_reviewing`。
- 超出 `max_fix_rounds` 后进入 `failed` 或 `blocked`。

验收标准：

- e2e 测试覆盖一次 request changes 后修复成功。
- e2e 测试覆盖 max fix rounds exhausted。
- 新 commit head sha 必须导致旧 review/check 失效。

建议标签：`type:feature`、`area:runtime`、`area:agents`、`priority:p1`

依赖：M8-01、M8-02、M9-01

需要更新文档：

- `docs/contracts/task-state-contracts.md`
- `docs/contracts/schemas/fix-result.schema.json`

### Issue M9-03：实现 CI failure / pending 的可恢复处理

建议标题：`M9: support recoverable CI pending and failed check handling`

背景：

- 当前 full lifecycle 对 required checks 偏同步判断；真实 GitHub Actions 常见 pending、delayed、failed 后重跑。

范围：

- `checks.pending` 保持 `ci_waiting`，不立即失败。
- `checks.failed` 进入 `fixing` 并触发 implementer 修复。
- 支持后续 check_run / workflow_run webhook resume。
- 记录 check evidence。

验收标准：

- pending check 不触发 merge。
- failed check 触发 fix loop。
- 后续 succeeded webhook 能继续 merge path。
- stale head check 不推进状态。

建议标签：`type:feature`、`area:ci`、`area:runtime`、`priority:p1`

依赖：M9-02

需要更新文档：

- `docs/contracts/task-state-contracts.md`
- `docs/api-design/03-internal-module-interfaces.md`

### Issue M9-04：补齐 GitHub 写成功但 SQLite 记录失败的恢复测试

建议标题：`M9: add crash-recovery tests for GitHub writes before SQLite action records`

背景：

- Codex 报告指出 REST adapter 的 idempotency 主要在调用侧，写成功后本地记录失败需要 reconciliation 修复。

范围：

- 为 comment、review、PR、merge、close issue 等写动作增加故障注入测试。
- 确认重放时通过 marker / deterministic lookup 修复本地状态，不重复写。
- 必要时调整 action record 顺序或增加 pending action 模型。

验收标准：

- 每类关键 GitHub write 都有“远端成功、本地中断、reconcile 修复”的测试。
- 重放不重复写评论/review。
- 冲突写入进入 `IDEMPOTENCY_CONFLICT` 或 blocked。

建议标签：`type:test`、`area:reconciliation`、`area:idempotency`、`priority:p1`

需要更新文档：

- `docs/contracts/data-contracts.md`
- `docs/contracts/github-artifact-templates.md`

### Issue M9-05：引入最小调度/重试循环

建议标题：`M9: add minimal scheduler for recoverable workflow states`

背景：

- 当前 runtime 更像一次 webhook 触发的一次同步推进。
- 长期运行需要处理 pending、lease expiry、agent retry、rate limit retry。

范围：

- 定义 recoverable states 的调度策略。
- 支持单进程周期性 scan 或显式 `ao reconcile --apply`。
- 尊重 pause、blocked、needs-human、lease。
- 不引入复杂队列系统。

验收标准：

- pending/fixing/lease expired run 可被恢复处理。
- paused/blocked/terminal 不被误推进。
- rate limited / agent process failed 可按策略重试。

建议标签：`type:feature`、`area:scheduler`、`priority:p1`

依赖：M9-01、M9-02、M9-03

需要更新文档：

- `docs/contracts/task-state-contracts.md`
- `docs/api-design/05-cli-commands.md`

## 5. Milestone M10 - Quality Gates And Verification

目标：把“测试通过”升级为“质量可视化、失败路径可回归、UI 可浏览器验证、release 可持续验证”。

### Issue M10-01：引入覆盖率统计和最小阈值

建议标题：`M10: add coverage reporting and baseline thresholds`

背景：

- Trae 报告指出缺少覆盖率指标，无法量化质量。

范围：

- 使用 Node test coverage 或轻量覆盖率工具。
- 建立当前 baseline。
- 对关键模块设置初始阈值，不追求一次性高阈值。
- CI 上传或打印 coverage summary。

验收标准：

- `npm run coverage` 可运行。
- CI 能看到行覆盖率和分支覆盖率。
- 阈值失败会阻断。

建议标签：`type:test`、`area:quality`、`priority:p1`

### Issue M10-02：补齐失败路径端到端测试矩阵

- e2e 覆盖 PR reviewer request changes。
- e2e 覆盖 checks failed / pending。

验收标准：

- 每类失败路径都有端到端测试。

建议标签：`type:test`、`area:runtime`、`priority:p1`

范围：

### Issue M10-04：补齐安全脱敏和 marker 独立测试

验收标准：

### Issue M12-02：拆分 runtime lifecycle 子流程
