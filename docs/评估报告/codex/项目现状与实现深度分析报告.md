## 1. 总体结论

AgentOrchestrator 已经不是只有方案文档的原型仓库。当前实现具备一个可运行的 GitHub-native Orchestrator 本地服务和 CLI 边界：可以校验本地配置、启动 webhook 服务、校验 GitHub webhook 签名、去重 delivery、规范化 GitHub 事件、维护 SQLite run 状态、调用外部 coding agent 进程、通过 GitHub REST adapter 执行评论/分支/提交/PR/review/check/merge/close issue 等操作，并提供本地只读 UI 查看 SQLite 调度状态。

项目的工程成熟度处在“可演示、可本地运行、已有一次 live smoke 记录，但核心自动化仍偏 happy path”的阶段。它已经有较完整的合同、schema、单元测试和 CLI 表面；但距离长期无人值守处理真实仓库 Issue，还有明显差距，主要集中在调度恢复、实际 diff/policy enforcement、失败修复循环、真实外部 agent 可控性、安装入口一致性和文档一致性。

一句话判断：当前实现适合继续作为 GitHub-native 自动处理 Issue 的 MVP 基座，不应直接宣传为稳定的生产级自治工程师系统。

## 4. 当前实现架构

### 4.1 CLI 和服务入口

`package.json` 定义：

- Node 要求：`>=26.0.0`
- bin：`ao -> ./src/cli.ts`
- 主校验：`npm run check`
- CLI 开发入口：`npm run cli -- <command>`
- e2e smoke：`npm run smoke:e2e`
- release binary 构建：`npm run build:sea`

`src/cli.ts` 是当前最大入口模块，承担：

- 命令解析。
- 配置读取与校验。
- `serve` runtime 启动。
- webhook request 处理。
- live/mock GitHub adapter 构造。
- lifecycle dispatch context 构造。
- `validate`、`doctor`、`live-check`、`live-smoke`、`reconcile`、`inspect-run`、`ui` 等命令。

当前 CLI 表面已经基本覆盖本地开发、live 前置检查、服务运行、状态检查、UI 查看和 smoke 验证。

### 4.2 Webhook intake

`src/orchestrator/webhook-runtime.ts` 实现了第一段确定性 intake：

- 仅接受 `IssueAutopilotRequested` 进入规划。
- 创建或读取 stable run id。
- 在 `new` 状态获取 lease。
- CAS 转换到 `planning`。
- 写 planning-started comment。
- 记录 idempotent action。

`src/cli.ts` 的 `startServeRuntime` 提供：

- `GET /healthz`
- `POST /webhook`
- webhook secret 缺失时返回明确错误。
- 原始 body 签名校验。
- delivery id 去重。
- GitHub payload 归一化为 domain event。
- 可选 lifecycle 模式下进入 `dispatchIssueWork`。

这个边界清晰，适合作为 webhook server MVP。

### 4.3 完整生命周期推进

`src/orchestrator/runtime-lifecycle.ts` 串起了完整 happy path：

1. 进入 planning。
2. 调 planner agent。
3. 写 plan comment。
4. 调 plan reviewer。
5. 写 plan review comment。
6. 调 implementer。
7. 创建分支。
8. 提交 changed files。
9. 创建或更新 PR。
10. 调一个或多个 PR reviewer。
11. 读取 required checks。
12. 评估 merge gate。
13. 调 merge API。
14. 删除分支。
15. 写 final summary。
16. 关闭 Issue。
17. 状态进入 `issue_closed`。

这说明项目已经具备真实“端到端编排路径”，而不只是离散工具函数。

但该路径仍偏一次性同步推进：

- 对长期运行队列、异步 CI 等待、失败后重试调度、服务重启恢复的支持有限。
- `runIssueLifecycleFromStep` 的恢复路径会构造 stub plan review 和 stub implementation，继续执行 CI/merge/closeout；这能让 resume smoke 跑通，但审计严谨性弱于“从 GitHub artifact 重新解析真实证据”。
- PR review 不通过和 CI 不通过时，当前代码主要转入 `fixing` 或抛错，还不是一个完整的自动修复、重新实现、重新 review、重新 CI 的长期循环。

### 4.4 GitHub adapter

`src/github/api.ts` 定义了清晰的 GitHub API adapter interface，包含：

- issue comment
- labels
- branch
- commit
- PR
- PR review
- check summary
- merge
- delete branch
- close issue

`src/github/rest-github-api.ts` 已实现真实 REST adapter：

- 使用 GitHub App token provider 获取 installation token。
- commit 前检查 branch 当前 head 是否等于 expected head。
- merge 时传入 expected head sha。
- check summary 同时读取 check runs 和 commit statuses。
- 401/403/404/409/422/429 映射到注册错误码。

这是一个正确的接口方向。限制是 REST adapter 自身没有全局持久化幂等能力，真正的幂等依赖调用侧传入 idempotency key 并记录到 SQLite。

### 4.5 Agent adapter

`src/agents/process-agent-adapter.ts` 的设计合理：

- 外部进程通过 stdin 接收 `{ envelope, prompt }`。
- stdout 必须输出 JSON。
- 支持 `_agent_meta` 包装，保留 agent/model attribution。
- 根据 role 校验输出 schema。
- 过滤 GitHub/token/secret/private/webhook/installation 等敏感环境变量。
- 默认超时 10 分钟，stdout/stderr 限制 1 MiB。

这个实现符合“agent 不拿 GitHub token”的安全边界。

仍需注意：

- 当前 prompt 文本非常短，例如 `Create a low-risk plan.`、`Implement the approved plan.`。真实 coding agent 是否能稳定产出符合 schema 且真正修改工作区，需要更多系统提示、上下文构造和失败恢复策略。
- agent 权限中的 `network`、`mode` 主要是配置语义，不等于 OS 级沙箱或进程级权限隔离。

### 4.6 SQLite 状态层

`src/state/sqlite-store.ts` 实现了：

- migrations
- workflow run snapshot
- lease acquisition
- CAS state update
- idempotent action record
- head invalidation
- artifact repair

`src/state/sqlite-queries.ts` 提供 UI/inspect 读模型：

- run list
- stats
- deliveries
- run detail

状态层设计符合合同层要求，测试覆盖也较好。

风险点：

- 主 lifecycle 在进入规划后没有在每个物料动作前显式重新 acquire/renew lease。当前更像单进程同步执行；如果未来多 worker 或长时间 agent 执行，lease 语义需要增强。
- 有些生命周期写动作没有调用统一的“先查 idempotent_actions 再跳过”的执行包装，而是依赖 adapter 和 record 后置记录。异常中断时的 replay 行为需要更多真实故障测试。

### 4.7 本地 UI

`src/ui/server.ts` 和 `src/ui/public/*` 已实现只读 UI：

- 默认 `127.0.0.1:23847`
- 拒绝 `0.0.0.0` 和 `::`
- 健康检查返回 `agent-orchestrator-ui`
- 提供 stats、runs、run detail、deliveries API
- 静态页面包括 dashboard、runs、run detail、deliveries
- 10 秒自动刷新
- UI 使用中文标题和英文状态词汇组合

这是一个有实用价值的 operator surface，适合本地调试。当前范围没有做写操作、GitHub enrichment、auth，这是合理边界。

### 4.8 Release binary

`.github/workflows/release.yml` 和 `tools/build/build-sea.mjs` 已加入 Node SEA 构建：

- Linux x64
- Windows x64
- macOS arm64
- macOS x64

构建脚本会 bundle CLI，并把 UI 静态资源嵌入 SEA asset。方向正确。

需要注意：

## 6. 关键问题与风险

### P0/P1：实际路径策略未集成到 full lifecycle

`src/policy/path-policy.ts` 有 `evaluatePathPolicy`，测试也覆盖 allow/deny/high-risk。但 CodeGraph 搜索显示该函数当前只被测试引用，没有被 `runtime-lifecycle` 调用。

影响：

- implementer 返回的 `changed_files` 会被 `runtime-lifecycle` 用来读取文件并提交。
- deny/high-risk/outside allow 的实际 diff 没有在 full lifecycle 中被强制拦截。
- 这与合同层“Actual changed files come from git, not agent output”和“deny/high-risk block automatic merge”存在偏差。

建议优先修复：

- 在 implementer 完成后，从 git diff 收集实际 changed files，而不是信任 agent result。
- 调用 `evaluatePathPolicy`。
- deny/high-risk/outside allow 进入 `blocked` 并写明原因。
- 测试覆盖 full lifecycle 中 deny/high-risk 文件不能提交 PR/merge。

### P1：Workspace manager 未进入 runtime 主路径

`src/workspace/manager.ts` 有 branch/path/diff 能力，但当前主 lifecycle 没有使用它创建隔离 worktree，也没有从实际 git diff 取证。

影响：

- agent 工作区隔离不足。
- changed_files 可信度不足。
- 本地 checkout 可能被 agent 直接修改，生命周期再按 agent 输出提交。

建议：

- full lifecycle 的 implementer 阶段必须通过 workspace manager 准备受控工作区。
- diff collection 必须作为 policy enforcement 的唯一文件证据。

### P1：恢复路径使用 stub 证据，审计严谨性不足

`runIssueLifecycleFromStep` 在 resume 路径构造 stub plan review 和 stub implementation，然后继续 CI/merge/closeout。

影响：

- 恢复后的 final summary、risk、tests、branch 可能不是从真实 GitHub artifact 或 SQLite action 重新解析得到。
- 对审计、可重放和事故排查不够严谨。

建议：

- 恢复路径应从 GitHub artifact markers、PR body、reviews、checks 和 SQLite actions 重建真实上下文。
- 缺证据时进入 `blocked`，不要构造可合并的 stub。

### P1：失败修复循环还不是完整生产循环

代码中有 fix-loop decision 和状态转换，但 full lifecycle 中遇到 PR review changes 或 checks failed 主要是抛错/转状态，不是完整地重新调用 implementer 修复、push 新 head、使旧 review/check 失效、重新 review、重新 CI。

影响：

- 真实仓库 CI 异步失败后需要人工或外部调度再触发。
- “最多 3 轮自动修复”的合同还未完全落成长期运行行为。

建议：

- 引入 queue/scheduler 或 event-driven resume loop。
- 对 `fixing -> pr_reviewing` 提供完整实现和测试。
- 将 CI pending/failure 作为可恢复状态，不应只依赖一次同步调用。

### P1：全局 `ao` 安装入口当前存在本机异常

本轮直接调用 `ao --help`、`ao validate`、`ao ui` 未在 10 秒内返回，已中断；但 `npm run cli -- ...` 正常。`ao` 当前解析到 Homebrew 全局 symlink：

```text
/opt/homebrew/bin/ao -> ../lib/node_modules/agent-orchestrator/src/cli.ts
```

影响：

- README 推荐的全局 `ao` 对当前机器不稳定。
- 可能影响用户按文档操作。

建议：

- 复查 npm link 后 shebang、Node 版本、stdin/TTY 行为。
- 在 README 增加“若全局 `ao` 异常，使用 `npm run cli --` 或 release binary”的明确故障处理。
- 将 `ao --help` 直接入口纳入 `npm run check` 或单独 smoke。

### P2：README 引用不存在的 repo policy 示例

当前仓库没有 `config/repo-policy.example.json`，但 README/使用说明中存在类似目标仓库策略配置示例的预期。

影响：

- 新用户缺少可复制的目标仓库 `.github/agent-orchestrator.json` 样例。

建议：

- 增加 `config/repo-policy.example.json` 或调整 README 指向真实存在的 fixture。
- 最好提供一份最小 low-risk policy 和一份更严格 production policy。

### P2：M6/M7 Done 与生产就绪之间容易被误读

进度台账写明 M6 live integration 和 M7 local UI 完成，这对里程碑管理没问题；但从工程风险看，它仍是 MVP 完成，不等于生产稳定。

建议：

- 在 README 或 progress 中新增“Production readiness gaps”。
- 明确列出：长期调度、真实 diff enforcement、fix loop、真实 agent prompt pack、部署监控、限流/backoff、密钥轮换、webhook redelivery recovery。

### P2：REST adapter 的 idempotency 主要在调用侧，不是 adapter 内建

adapter method 都要求 idempotency key，但 GitHub REST API 本身没有这些 key。当前实现依赖 SQLite 记录和 adapter 的局部幂等逻辑。

影响：

- 如果进程在 GitHub 写成功但 SQLite record 前崩溃，需要 reconciliation 正确修复，否则可能重复写评论/review。

建议：

- 对每类 GitHub artifact 都通过 marker 或 deterministic branch/PR 查询实现 replay-safe。
- 增加“GitHub 写成功、SQLite 记录失败”的故障注入测试。

## 7. 与原始方案的符合度

符合较好的部分：

- GitHub-native 事实来源。
- GitHub App + webhook + REST API。
- label-gated autopilot。
- 多 agent 角色抽象。
- SQLite 最小状态。
- head sha invalidation。
- merge API 最终 gate。
- 本地 UI 不作为第一版核心控制台。

部分符合但仍不完整：

- 自动修复 review/CI 问题：已有状态和决策，但完整循环不足。
- 可追踪、可重放：有 marker、SQLite actions、reconciliation，但恢复路径仍有 stub。
- policy enforcement：path policy 有实现和测试，但 full lifecycle 未强制调用。
- agent 间接协作：接口成立，但 prompt/context 仍偏简化。

尚未达到的生产级能力：

- 长期后台调度和任务队列。
- 多 worker 下严格 lease renewal 和动作独占。
- CI pending 的异步等待/回调恢复。
- 部署级 observability。
- 对真实 coding agent 输出失败的系统化 retry/backoff。
- 真实仓库复杂 diff 的安全提交策略。

## 8. 建议的下一步优先级

### 第一优先级：把 policy 和 workspace manager 接进 full lifecycle

目标：让主路径不再信任 agent 输出的 changed_files，而是信任 git diff。

验收：

- deny path 不会被提交。
- high-risk path 自动 blocked。
- outside allow path 自动 blocked。
- final PR body/summary 使用实际 diff evidence。

### 第二优先级：补齐真实恢复证据

目标：resume 不再构造 stub plan/review/implementation。

验收：

- 从 GitHub comments/PR body/reviews/checks 解析 marker 和 head sha。
- 缺少关键 artifact 时进入 blocked。
- 断点恢复后 final summary 使用真实 plan/implementation/risk/tests。

### 第三优先级：实现完整 fix loop

目标：PR review changes 或 CI failed 能自动进入 implementer 修复，并产生新 commit/head。

验收：

- `pr_reviewing -> fixing -> pr_reviewing` 完整跑通。
- `ci_waiting -> fixing -> pr_reviewing` 完整跑通。
- 新 head 使旧 review/check/merge-ready 失效。
- 超过 max fix rounds 后进入 failed 或 blocked。

### 第四优先级：整理安装和示例

目标：降低新用户按 README 启动的失败率。

验收：

- `ao --help` 全局入口 smoke 稳定。
- 提供真实存在的 repo policy 示例。
- README 区分源码入口、npm link 入口、release binary 入口。

### 第五优先级：生产运行边界说明

目标：防止 Done 台账被误解为生产稳定。

验收：

- README 增加“当前适用场景”和“尚不适合场景”。
- 进度文档新增 production readiness gap 清单。

## 9. 结论

当前项目已经形成了可运行的 MVP：CLI、webhook server、SQLite 状态、GitHub App auth、REST adapter、agent process adapter、mock e2e、local UI 和 release binary 工作流都存在，并且测试基础扎实。

真正需要警惕的是不要被“全生命周期 smoke 通过”和“进度全 Done”误导。当前主路径仍然偏同步 happy path，关键安全控制中的实际 diff/policy enforcement 尚未接入主生命周期，恢复路径也有 stub 证据。下一轮开发应优先把 workspace manager、path policy、真实 artifact 恢复和 fix loop 接入运行主线。完成这些后，项目才更接近可以长期处理真实 GitHub Issue 的自治编排器。
