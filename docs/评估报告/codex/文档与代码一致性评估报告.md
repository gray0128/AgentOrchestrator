# 文档与实际代码一致性评估报告

评估日期：2026-06-27

评估对象：

- `github-native-agent-orchestrator-自动处理-issue-方案.md`
- `docs/contracts/`
- `docs/api-design/`
- `src/`、`test/`、`tools/` 下当前实现

评估边界：

- 本报告只核对仓库内设计文档、契约/API 文档与当前代码的一致性。
- 未核对实时 GitHub issue、PR、ruleset、branch protection 或线上运行状态。
- 仓库已启用 CodeGraph，代码定位先通过 CodeGraph，再针对关键文件回读核对。

## 结论摘要

当前代码已经从早期骨架推进到可运行的本地 Orchestrator MVP：CLI、webhook intake、签名校验、delivery 去重、SQLite 状态库、agent process adapter、worktree/diff 校验、PR review/fix loop、check aggregation、merge/closeout、本地只读 UI 都有实现和测试覆盖。

总体判断：**主架构和大多数 contract 已基本对齐，但仍有若干实现缺口与文档漂移。**

最重要的差异有四类：

1. **Webhook 事件覆盖不足**：主方案列出的 `issues.opened`、`pull_request_review.submitted`、`status` 未归一化；`repository_ruleset` / `branch_protection_rule` 也未实现。
2. **Query API 抽象未落地**：`GitHubApi.getIssueContext`、`getPullRequestContext`、`PolicyEngine.evaluateAutopilot` 等文档接口没有同名实现，能力分散在 lifecycle 入参构造、artifact reader、check summary 和 actor gate 中。
3. **Merge gate 预检查不完整**：代码会用 GitHub Merge API 携带当前 head sha 做最终 gate，但生命周期内传给 `evaluateMergeGate` 的 `githubMergeable` 是固定 `true`，没有按文档要求在 merge gate 前重新读取 PR mergeability / ruleset / branch protection 证据。
4. **idempotent_actions 审计不完整**：SQLite schema 与 contract 存在，但 full lifecycle 中 branch、commit、PR、PR review、merge、delete branch、close issue 等多类写动作虽然传入 idempotency key，却没有全部写入本地 `idempotent_actions` 表。

## 主要不一致与缺口

### 1. 主方案仍写“MVP 不做本地 Web UI”，但 API 文档和代码已经实现 `ao ui`

主方案 `2.3 边界` 写明 “MVP 不做：不做本地 Web UI”。但 `docs/api-design/README.md` 和 `08-local-ui-api.md` 明确把 `ao ui` 纳入 MVP 本地 operator surface，代码也已经实现。

判断：这是**文档演进不一致**，不是代码缺陷。建议更新主方案边界，区分：

- 不做“完整用户任务控制台/写操作 UI”；
- 已做“localhost 只读 SQLite operator UI”。

### 2. Webhook 管理 API 路径不一致

`docs/api-design/04-management-apis.md` 写入站操作为 `POST /webhooks/github`，但 `startServeRuntime()` 实际只注册：

- `GET /healthz`
- `POST /webhook`

`ao live-smoke` 也向 `/webhook` 发送请求，因此实现内部是一致的，滞后的是管理 API 文档。

建议：将 `04-management-apis.md` 改为 `/webhook`，或在代码中兼容 `/webhooks/github`。短期建议优先改文档，因为 CLI 和测试都已经围绕 `/webhook`。

### 3. Webhook 事件覆盖低于主方案清单

主方案 `9.1 Webhook Events` 要求处理：

- `issues.opened`
- `issues.labeled`
- `issues.unlabeled`
- `issue_comment.created`
- `pull_request.opened`
- `pull_request.synchronize`
- `pull_request.reopened`
- `pull_request_review.submitted`
- `check_run.completed`
- `status`
- `workflow_run.completed`

当前 `normalizeGitHubWebhook()` 支持：

- `issues.labeled`：`agent:autopilot`、`agent:pause`、`agent:no-merge`
- `issues.unlabeled`：仅 `agent:pause`
- `issue_comment.created`：mention dispatch
- `pull_request_review_comment.created`：mention dispatch，属于扩展能力
- `pull_request.synchronize`
- `check_run`
- `workflow_run`

缺口：

- `issues.opened` 不会因已有 `agent:autopilot` label 触发。
- 移除 `agent:autopilot` 没有取消或停止推进事件。
- `pull_request.opened` / `reopened` 未归一化。
- `pull_request_review.submitted` 未归一化，人工 review 不能直接驱动状态。
- legacy `status` webhook 未归一化，虽然 REST adapter 会读取 combined status。
- 可选 `repository_ruleset` / `branch_protection_rule` 未实现。

影响：自动化主路径可通过 label/mention/lifecycle 运行，但与“GitHub 事件驱动恢复/推进”的设计覆盖面不完全一致。

### 4. Query API 文档接口没有同名实现

`docs/api-design/02-query-apis.md` 定义了：

- `GitHubApi.getIssueContext`
- `GitHubApi.getPullRequestContext`
- `StateStore.getRunForIssue`
- `PolicyEngine.evaluateAutopilot`
- `PolicyEngine.evaluateMergeGate`

当前实现：

- `GitHubApiAdapter` 是写入和 check summary 接口，没有 `getIssueContext` / `getPullRequestContext`。
- GitHub artifact 读取存在于 `src/reconciliation/github-artifacts.ts`，但不是统一 Query API。
- actor/autopilot gating 分散在 `shouldDiscardActor()`、repo policy loading 和 `buildDispatchContext()`。
- `evaluateMergeGate()` 已实现，但输入由 lifecycle 组装。

判断：这是**架构接口未收敛**。能力不是完全缺失，但代码没有按 API design 的模块边界实现。

### 5. Merge gate 预检查没有完整使用 GitHub 当前 PR 上下文

文档要求 merge gate 前重新读取 Issue、PR、labels、reviews、check runs、combined status、当前 `head_sha`、mergeability、ruleset/branch protection evidence。

当前实现中：

- checks 会通过 `readCheckSummary()` 读取当前 head 的 check runs 和 combined status。
- PR reviewer verdict 与 check aggregation 会绑定 head sha。
- `mergePullRequest()` 会携带 `expectedHeadSha`，GitHub Merge API 是最终 gate。

但：

- `finishMergeAndCloseout()` 调用 `evaluateMergeGate()` 时传入 `githubMergeable: true`。
- 没有在 merge gate 前统一重新读取 PR `mergeable` / `mergeable_state`。
- 没有读取 branch protection / ruleset 证据；`checks.source = branch_protection_read` 只是 schema 枚举和 validation 接受。
- 没有将“GitHub 正在计算 mergeability”映射为 `WAIT`。

判断：**最终安全性依赖 GitHub merge API，符合硬约束；但 deterministic merge gate 的预检查与审计证据不足。**

### 6. `idempotent_actions` 本地审计没有覆盖所有写动作

文档要求每个 GitHub 写动作都进入 `idempotent_actions`，并用 stable idempotency key 防重放。

当前 full lifecycle 里确实给 GitHub adapter 传了 idempotency key，但本地只记录了部分动作，例如：

- planning started comment
- triage comment
- plan comment
- plan review comment
- check summary read
- fix comment
- blocked comment / labels

未看到 full lifecycle 对这些动作统一写入 `idempotent_actions`：

- `createBranch`
- `commitChanges`
- `createOrUpdatePullRequest`
- `submitPullRequestReview`
- `mergePullRequest`
- `deleteBranch`
- final summary comment
- `closeIssue`

影响：

- SQLite 审计表无法完整复盘实际 GitHub 写动作。
- crash 后“远端写成功、本地动作未记录”的恢复依赖 artifact reader/backfill，但普通 lifecycle 本身没有先写 pending/action 再提交 completed。
- 与 `data-contracts.md`、`04-management-apis.md` 的幂等写动作 contract 不完全一致。

### 7. 状态 label 互斥能力存在，但主生命周期没有全面同步状态 labels

`src/state/labels.ts` 实现了 `syncStateLabels()`，测试覆盖互斥状态 label。但 full lifecycle 的常规状态推进主要更新 SQLite，并不会在每个状态进入时调用 `setIssueLabels()` 同步 `agent:planning`、`agent:plan-review`、`agent:implementing`、`agent:pr-review`、`agent:merge-ready`、`agent:done`。

已同步的主要是 blocked 场景。

影响：GitHub Issue label 作为用户界面/恢复线索的承诺尚未完全兑现。

### 8. PR review 事件与 agent review 模型存在边界差异

代码内置 PR reviewer agent，并通过 `submitPullRequestReview()` 写 GitHub PR review。多 reviewer 配置和 `required_pr_approvals` 也已有实现。

但文档同时描述 GitHub `pull_request_review.submitted` 事件驱动状态。当前实现没有读取外部 review webhook，因此：

- agent 自产 review 能推进内联 lifecycle；
- 外部人工 review 不能通过 webhook 直接推进状态；
- 如果仓库 branch protection 需要人工 review，当前只能依赖 GitHub Merge API 在最终 merge 时拒绝，而不是 deterministic gate 提前等待。

## 风险分级

### 高风险

1. **idempotent_actions 不完整**
   - 原因：影响幂等、审计和 crash recovery。
   - 建议：统一封装 GitHub write executor，按 `pending -> completed/failed/skipped` 写动作记录，再调用 adapter。

2. **merge gate 预检查不足**
   - 原因：虽然 Merge API 是最终 gate，但缺少文档要求的 mergeability/ruleset/branch protection 证据和 WAIT 语义。
   - 建议：实现 `getPullRequestContext` 或等价 query 层，把 PR mergeability、reviews、current head、checks 统一交给 `evaluateMergeGate()`。

### 中风险

1. **Webhook coverage 不完整**
   - 建议优先补 `pull_request_review.submitted`、`status`、`issues.opened`、`agent:autopilot` removed。

2. **Query API 边界未收敛**
   - 建议将 artifact reader、check summary、issue/pr context 读取整合到文档命名的 query adapter，减少 lifecycle 直接拼上下文。

3. **状态 label 未全流程同步**
   - 建议在状态 transition side effects 中统一调用 `syncStateLabels()`，并记录 label write action。

### 低风险

1. **主方案边界未同步 `ao ui`**
   - 建议文档修正即可。

2. **管理 API webhook path 文档漂移**
   - 建议文档修正或代码兼容双路径。

## 建议修复顺序

1. 先修文档漂移：主方案 Web UI 边界、`04-management-apis.md` webhook path。
2. 抽象统一 GitHub write action executor，补齐 `idempotent_actions` 记录。
3. 收敛 Query API：实现或重命名 `getIssueContext`、`getPullRequestContext`，让 merge gate 输入来自统一当前 GitHub 事实读取。
4. 补齐 webhook events：`status`、`pull_request_review.submitted`、`issues.opened`、`issues.unlabeled agent:autopilot`。
5. 将 `syncStateLabels()` 接入每个对用户可见的状态 transition。
6. 为 `checks.source = branch_protection_read` 明确实现、禁用或降级行为；避免 schema 接受但 runtime 忽略。
